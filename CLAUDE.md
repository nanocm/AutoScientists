# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

AutoScientists is a multi-agent system where Claude Code subagents self-organize into teams to run long-horizon scientific experiments. Agents coordinate through a local [ClawInstitute](https://www.npmjs.com/package/clawinstitute) server (HTTP API at `http://localhost:3000/api/v1`) using workshops (group discussion boards), workspaces (versioned shared filesystems), and posts (structured messages).

The **orchestrator** is a pure coordinator: it launches agents and harvests results but never runs training itself. The **agents** (GPU runners, analysts, monitor) do all the scientific work.

## Prerequisites and Setup

```bash
# Start the coordination server (required before any launch)
npx clawinstitute start

# Install Python deps
pip install -r requirements.txt   # requests, pyyaml
```

The server must be running whenever agents execute. It stores workshops, workspaces, posts, and notifications in memory (or persisted to disk depending on configuration).

## Running an Experiment

```bash
# Syntax: claude -p "Read runbook.md and execute. Task: <task-path>. Run name: <name>."
claude -p "Read runbook.md and execute. Task: task-autoresearch. Run name: ar_v1."
claude -p "Read runbook.md and execute. Task: task-biomlbench/drug_discovery/tdcommons-lipophilicity-astrazeneca. Run name: lipo_v1."
claude -p "Read runbook.md and execute. Task: task-protein-gym. Run name: spike_v1."
```

Each launch creates `../<run-name>/` as a sibling of the template — the template stays clean. You can also run `launch.py` directly for more control:

```bash
python3 launch.py <run-name> --task task-autoresearch
python3 launch.py <run-name> --task task-biomlbench/drug_discovery/tdcommons-lipophilicity-astrazeneca
python3 launch.py <run-name> --task task-protein-gym --protein SPIKE_SARS2_Starr_2020_binding
python3 launch.py <run-name> --task <path> --output-dir /tmp/runs
```

`--task` must point at a leaf directory that has `TASK.md`. Passing a family directory (e.g. `task-biomlbench`) without a specific subtask is an error.

## Architecture

### Control Flow

```
runbook.md           Universal orchestrator program (hook-based, task-agnostic)
  └── task-profile.md  Task-specific hooks (copied from LAUNCH.md by launch.py)
```

`runbook.md` defines the universal loop: bootstrap → discussion → form teams → execution loop (analysts propose, GPU agents run experiments, champion promoted on improvement). Every variation point is a **hook** that `task-profile.md` fills in. When `runbook.md` says `→ PROFILE HOOK: <name>`, the orchestrator reads the corresponding `## Hook: <name>` section in `task-profile.md` and executes it.

### Agent Roles (10 agents per run)

| Agent | Count | What it does |
|---|---|---|
| `{prefix}_monitor` | 1 | Bootstrap, team formation, health checks |
| `{prefix}_gpu{1-6}` | 6 | Claim experiments, run training, record results |
| `{prefix}_analyst{1-3}` | 3 | Research, propose experiments, prune dead ends |

Each agent gets its own directory under `agents/<name>/` with credentials, `AGENT.md` (identity/state), `HEARTBEAT.md` (complete per-cycle protocol), and `memory/`.

### Agent Boot: HEARTBEAT.md

Every agent starts by reading its `HEARTBEAT.md` and executing **Part 0 (Mode Selector)** first — this routes to one of five branches:

- **Part 2** — Discussion mode (`MODE=discussion` set, or active `[DISCUSSION-TRIGGER]` post, or empty roster)
- **Part 3** — No-team exit (teams exist but agent wasn't assigned)
- **Part 4** — Normal cycle (has a team, `MODE=execute`)
- **Part 5** — Resume-and-post (GPU agent with an unposted prior result)
- **Resume-waiting** — GPU agent whose training subprocess is still alive

**Never use `haiku`-class models for analysts or GPU agents.** Haiku has a documented failure mode where it writes local memory files claiming completion without ever calling the ClawInstitute API.

### Run Directory Layout

```
<run-name>/
├── runbook.md           Orchestrator program (copied from template)
├── task-profile.md      Task hooks (copied from task's LAUNCH.md)
├── task/                Task definition and data (never write train.py/submission.csv here directly)
├── system/              Agent role templates and reference docs
├── agents/<name>/       Per-agent directories
│   ├── HEARTBEAT.md     Agent's complete protocol (generated by launch.py)
│   ├── AGENT.md         Agent state (role, team, last session)
│   ├── credentials.json API token
│   ├── memory/          Per-agent persistent memories
│   └── workspace/repo/  Agent's working copy for experiments
├── champion/            Current best solution (orchestrator-managed)
│   ├── train.py         Best training code (optimization/proteingym)
│   └── SOURCE           Provenance metadata
├── logs/
│   ├── experiments.jsonl  Canonical experiment log (orchestrator writes)
│   ├── sessions.jsonl     Per-agent-session log (orchestrator writes)
│   └── raw/               Agent stdout/stderr
├── WORKSPACE_ID         ClawInstitute main workspace UUID
├── WORKSHOP_NAME        Workshop name (e.g. `ar_my_run`)
├── agent_tokens.json    Per-agent API tokens
└── run_metadata.json    Run configuration snapshot
```

### ClawInstitute API Patterns

All agent coordination goes through the API. The API does **not** auto-parse YAML frontmatter — agents must parse it client-side:

```python
def parse_frontmatter(resp):
    content = resp.get("content", "")
    parts = content.split("---")
    return yaml.safe_load(parts[1]) if len(parts) >= 3 else {}
```

Key patterns:
- `PATCH /workspaces/{id}/files/{path}` with `{"frontmatter": {"key.subkey": value}}` for concurrent-safe partial updates (different keys don't conflict)
- `PUT` with `If-Match` header for optimistic locking (returns 409 on conflict)
- `GET /workspaces/{id}/files` (list metadata only) before reading content — never fetch every file blindly
- `X-Agent-Name` header is required on posts/comments to attach writer identity

### Workspace File Isolation Rule (biomlbench)

GPU agents must **never** write to `task/submission.csv` or `champion/train.py` directly. They save to their own agent workspace (`agents/{name}/workspace/repo/submission_{exp_id}.csv`) and write `result_latest.json`. The orchestrator is the only entity that promotes to canonical paths.

## Task Types

Three task types, set via `task_type:` in `TASK.md` frontmatter:

| `task_type` | Example | What agents evolve | Stop condition |
|---|---|---|---|
| `optimization` | `task-autoresearch` | `repo/train.py` (nanoGPT) | Stagnation (0 KEEPs in last 10) |
| `biomlbench` | `task-biomlbench/<category>/<task>` | `train.py` from scratch | Wall-clock deadline (8–16 h) |
| `proteingym` | `task-protein-gym` | `repo/kermut.py` | Stagnation |

### Adding a New Task

1. Create `task-<name>/TASK.md` with YAML frontmatter (`task_type`, `name`, metric direction)
2. Create `task-<name>/LAUNCH.md` with the 13 hook sections (`launch_command`, `discussion_policy`, `gpu_dispatch`, `champion_promotion`, `stagnation_response`, `exit_condition`, etc.)
3. Optionally add a data/repo setup script

`launch.py` walks up from the `--task` path to find the nearest `LAUNCH.md`, so a family-level `LAUNCH.md` (like `task-biomlbench/LAUNCH.md`) covers all subtasks unless overridden by a per-subtask one.

## Key System Invariants

- **Orchestrator never runs experiments.** It only launches agents, releases stale claims (>30 min, no result file), promotes champions, and logs.
- **Discussion-before-queuing rule.** Every experiment must start as a `[PROPOSAL]` post with ≥1 comment before entering the team queue.
- **`experiments.jsonl` is the canonical log.** The stagnation check (0 KEEPs in last 10) reads this file only.
- **Champion propagation.** Orchestrator copies winning `train.py` to `champion/train.py` after each KEEP. All GPU agents read from this canonical path.
- **Meta-improvement every 3 cycles** (optimization tasks). The orchestrator reads diagnostics from `experiments.jsonl` and makes exactly one targeted edit to a role template or task doc. Writing a report without changing a file does not count.
- **`sonnet` (or `opus`) for analysts and GPU agents.** Haiku fails silently in this workflow.

## Monitoring a Running Experiment

```bash
# Watch experiment outcomes
tail -f <run-name>/logs/experiments.jsonl | python3 -c "
import sys, json
for line in sys.stdin:
    d = json.loads(line)
    print(f'{d[\"completed_at\"]} {d[\"exp_id\"]:40s} {d.get(\"metric\",\"?\")} {d.get(\"outcome\",\"?\")} delta={d.get(\"delta\",\"?\"):+.6f}')
"

# KEEP rate per team
cat <run-name>/logs/experiments.jsonl | python3 -c "
import sys, json
from collections import defaultdict
stats = defaultdict(lambda: {'keep': 0, 'total': 0})
for line in sys.stdin:
    d = json.loads(line)
    stats[d['team']]['total'] += 1
    if d.get('outcome') == 'KEEP':
        stats[d['team']]['keep'] += 1
for team, s in stats.items():
    print(f'{team}: {s[\"keep\"]}/{s[\"total\"]} KEEPs ({100*s[\"keep\"]/max(s[\"total\"],1):.0f}%)')
"

# Champion history
grep '"outcome": "KEEP"' <run-name>/logs/experiments.jsonl | python3 -c "
import sys, json
for line in sys.stdin:
    d = json.loads(line)
    print(f'{d[\"completed_at\"]} {d[\"exp_id\"]:40s} metric={d.get(\"metric\",\"?\")} agent={d[\"agent\"]}')
"
```
