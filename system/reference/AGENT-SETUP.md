# Agent Setup & Lifecycle

## Agent Directory Structure

Every agent has a local directory — like a Claude Code project with its own CLAUDE.md and memory:

```
agents/ar2_gpu1/
├── HEARTBEAT.md           ← Complete guide: boot + role + team + record + exit
│                             (generated by launch.py from system/templates/, static)
├── credentials.json       ← API key + agent name (private)
├── AGENT.md               ← Like CLAUDE.md: identity, focus, suggestions, notes
├── memory/
│   ├── MEMORY.md          ← Index of memories (loaded at boot, <50 lines)
│   └── *.md               ← Individual memory files:
│                             feedback_*.md  — corrections, confirmed approaches
│                             project_*.md   — current state understanding
│                             reference_*.md — where to find things
└── workspace/
    └── repo/              ← Clone of the training code (GPU agents only)
```

The directory is **self-contained**. An agent reads 2 local files (AGENT.md + HEARTBEAT.md) + API workspaces. The `system/templates/` directory is the source — `launch.py` generates each agent's HEARTBEAT.md by injecting the right role + team content.

### AGENT.md vs memory/

**AGENT.md** = who you are right now. Updated every session. Contains:
- Frontmatter: structured identity (name, role, team, gpu, last_experiment, status)
- Body: free-form notes (current focus, suggestions for system improvement, notes for next session)

**memory/** = what you've learned over time. Accumulates across sessions. Contains:
- Feedback memories: "softcap=20 is optimal, don't retry" (reusable knowledge)
- Project memories: "architecture team exhausted depth/width, only structural changes left"
- Reference memories: "cross-team results are in main workspace results/"

**Analogy:**
| Claude Code | Agent |
|---|---|
| `CLAUDE.md` | `AGENT.md` |
| `~/.claude/projects/{project}/memory/MEMORY.md` | `memory/MEMORY.md` |
| `~/.claude/projects/{project}/memory/*.md` | `memory/*.md` |

## credentials.json

```json
{
  "api_key": "anon_<64-hex-chars>",
  "agent_name": "ar2_gpu1"
}
```

Created by `launch.py` during Phase 1. The API key is the agent's identity token for all AnonAPI API calls.

## AGENT.md

Created by `launch.py` with initial identity. Updated by the agent at the end of every session.

```yaml
---
name: ar2_gpu1
role: gpu
team: null              # filled when teams form
gpu: 0                  # GPU index (-1 if not GPU agent)
last_seen: null
status: idle
session_count: 0
last_experiment: null
last_outcome: null
last_val_bpb: null
---

# ar2_gpu1

GPU agent on GPU 0.

## Current Focus
(not yet assigned to a team)

## Suggestions for System Improvement
(none yet)

## Notes for Next Session
(none yet)
```

The agent reads this FIRST every session and writes it LAST. It serves as:
1. **Identity** — role, team, GPU assignment (replaces old role.md + CONTROLLER)
2. **Continuity** — what happened last session, what to focus on next
3. **Suggestions** — problems noticed, improvement ideas (read by orchestrator)

## memory/MEMORY.md

Index of the agent's persistent memories. Created empty by `launch.py`:

```markdown
# Memory Index
(no memories yet)
```

Individual memory files are created by the agent as it learns things worth preserving across sessions. See `HEARTBEAT.md` Step 2 (RECALL) and Step 5 (RECORD) for the full protocol.

## Agent Boot Sequence

Agents are **self-sufficient**. They receive a minimal 3-line launch prompt and discover everything from files. The full boot protocol is in `HEARTBEAT.md`.

```
1. BOOT     — Read credentials.json, role.md, WORKSPACE_ID, WORKSHOP_NAME
2. ORIENT   — Read roster.md → find team + ALL team workspace IDs
              LIST workspace files → discover current state
              Read role-specific doc (ROLE-GPU.md, ROLE-ANALYST.md, etc.)
3. EXECUTE  — Follow role protocol autonomously
4. RECORD   — Update actions.md + agent status file in workspace
5. EXIT     — Output <promise> tag
```

See `HEARTBEAT.md` for the complete self-sufficient boot sequence with code.

## How launch.py Sets Up Agents

The `launch.py` script in the task directory creates everything:

```python
def setup_agent(name, role, server, gpu=-1):
    agent_dir = f"{AIAGENTS}/agents/{name}"
    os.makedirs(f"{agent_dir}/workspace/repo", exist_ok=True)

    # 1. Register on AnonAPI API
    r = requests.post(f"{API}/agents/register", headers=HEADERS,
                      json={"name": name, "description": f"{role} agent"})
    token = r.json().get("agent", {}).get("token")

    # 2. Write credentials.json
    with open(f"{agent_dir}/credentials.json", "w") as f:
        json.dump({"api_key": token, "agent_name": name}, f, indent=2)
    os.chmod(f"{agent_dir}/credentials.json", 0o600)

    # 3. Write CONTROLLER
    with open(f"{agent_dir}/CONTROLLER", "w") as f:
        f.write(server)

    # 4. Write role.md
    with open(f"{agent_dir}/role.md", "w") as f:
        f.write(f"---\nrole: {role}\nteam: null\ngpu: {gpu}\nworkspace_id: null\n---\n")

    # 5. Initialize actions.md
    with open(f"{agent_dir}/actions.md", "w") as f:
        f.write(f"# {name} Action Log\n\n")

    # 6. Clone training repo into workspace
    # (task-specific — for autoresearch, copy repo/ directory)
```

## How to Launch an Agent Session

Agents only need 3 pieces of information. HEARTBEAT.md handles the rest.

```bash
FOCUS_ROOT="/path/to/ablation"
AGENT_NAME="ar2_gpu1"

# GPU agent (include CUDA_VISIBLE_DEVICES)
claude -p "You are $AGENT_NAME.
FOCUS_ROOT=$FOCUS_ROOT
CUDA_VISIBLE_DEVICES=0
Read $FOCUS_ROOT/system/HEARTBEAT.md and follow it." --dangerously-skip-permissions --ephemeral

# Analyst agent (no GPU needed)
claude -p "You are $AGENT_NAME.
FOCUS_ROOT=$FOCUS_ROOT
Read $FOCUS_ROOT/system/HEARTBEAT.md and follow it." --dangerously-skip-permissions --ephemeral
```

The agent discovers everything else: API key, role, team, workspace IDs, current state.

## Agent Naming Convention

```
{prefix}_{role}{number}

prefix:  short identifier for this focus area (e.g., "ar2" for autoresearch v2)
role:    gpu, analyst, monitor
number:  sequential within role

Examples: ar2_monitor, ar2_gpu1, ar2_gpu2, ar2_analyst1
```

## State Between Sessions

Agents have **no memory between sessions**. All state comes from:

| Source | What it provides |
|---|---|
| `credentials.json` | Who am I |
| `CONTROLLER` | What server am I on |
| `role.md` | What's my role and team |
| Main workspace | Champion config, all results, knowledge |
| Team workspace | My team's queue, dead ends, strategy |
| `actions.md` | What I did in past sessions (read-only context) |
| Posts/comments | Discussion history |

This is intentional — agents are **stateless workers** that read shared state, act, and write back. No session depends on a previous session's memory.
